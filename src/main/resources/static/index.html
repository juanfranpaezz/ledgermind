LedgerMind · demo de auditor

Core de pagos con ledger de doble entrada, idempotencia y auditoría post-cuántica vía MCP.

⚖️ DEMO con dinero simulado. El ledger es tamper-evident (append-only + hash/firma), no un sistema de prevención de fraude de producción. Lo real y defendible es el diseño de concurrencia e idempotencia que está debajo.  ·  Código

1 · Idempotencia: un reintento no cobra dos veces

Hago la misma transferencia dos veces con la misma idempotencyKey. Debe crear un solo asiento y mover la plata una sola vez.

—

2 · Tamper-evidence: editar el libro rompe la cadena

Simulo un atacante con acceso a la base que edita un asiento por SQL directo. La firma ML-DSA sigue válida (firmó la cabeza original), pero la hash-chain ya no recomputa: prueba de que lo alteraron después de firmar.

—

3 · El auditor responde: ¿el ledger fue manipulado?

Esto es el mismo informe que el tool MCP verify_journal_integrity le devuelve a un agente de IA. El agente lee y explica; nunca puede mover un peso.

—

4 · Conciliación: el agente explica los descuadres con el PSP

Cruzo el ledger contra un feed (simulado) de liquidación del PSP. El matching es determinista en Java; el agente solo narra cada descuadre: una comisión no asentada, un cobro del PSP sin asiento, un asiento que el PSP no reporta. La IA explica, el código decide.
